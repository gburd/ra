# PostgreSQL Genetic Query Optimizer (GEQO)

**Source:** https://www.postgresql.org/docs/current/geqo.html
**Date:** Reference documentation (current)
**Speaker:** Martin Utesch / PostgreSQL documentation

## Key Points
- GEQO activates for queries joining 12+ tables (configurable)
- Uses genetic algorithm to search join orderings
- Replaces exhaustive DP when search space is too large
- Non-deterministic: different ANALYZE runs may produce different plans

## GEQO Architecture

### Activation
- geqo_threshold = 12 (number of FROM items)
- When triggered, replaces standard DP join enumeration
- Can be disabled entirely: SET geqo = off

### Algorithm Parameters
| Parameter | Default | Description |
|-----------|---------|-------------|
| geqo_effort | 5 (1-10) | Planning time vs quality tradeoff |
| geqo_pool_size | auto | Population size (100-1000 typically) |
| geqo_generations | auto | Number of iterations |
| geqo_selection_bias | 2.0 | Selection pressure (1.5-2.0) |
| geqo_seed | 0 | Random seed for reproducibility |

### How It Works
1. Generate random population of join orderings
2. Evaluate fitness (plan cost) of each ordering
3. Select fittest individuals for reproduction
4. Crossover: combine parts of two parent orderings
5. Mutation: randomly modify join orderings
6. Repeat for geqo_generations
7. Return best ordering found

### Known Issues
- Non-deterministic: plan varies between runs
- No guarantee of optimality
- PostgreSQL TODO: investigate compressed annealing as replacement
- Poor for queries where specific join ordering is critical
- May miss good bushy trees (tends toward left-deep)

## Applicable to RA
- RA has no equivalent for handling large join graphs
- Gap: No genetic/evolutionary join ordering for large queries
- Gap: No simulated annealing alternative
- Gap: No compressed annealing (PostgreSQL TODO item)
- Gap: No randomized search fallback for egg saturation timeout
- Gap: No configurable search effort tradeoff

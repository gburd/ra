**Speaker:** Andy Pavlo

## Key Points
- Cost models translate physical plan structure into estimated execution cost
- Three components: CPU cost, I/O cost, memory cost (plus network for distributed)
- Cardinality estimation is the weakest link - errors compound exponentially
- Statistics freshness is critical and often neglected

## Cost Model Components

### Physical Cost Model
- Predicts CPU cycles, I/O operations, cache misses
- Hardware-dependent: needs calibration per deployment
- PostgreSQL approach: relative cost units (seq_page_cost = 1.0 baseline)
- DuckDB approach: instrumented micro-benchmarks for calibration

### Logical Cost Model
- Estimates intermediate result cardinalities
- Uses statistics: histograms, MCV lists, distinct counts
- Propagates cardinality through operator tree
- Key formulas:
  - Selection: output = input * selectivity
  - Join: output = |R| * |S| * selectivity
  - Aggregation: output = num_distinct_groups

### PostgreSQL Cost Parameters (calibration targets)
| Parameter | Default | Description |
|-----------|---------|-------------|
| seq_page_cost | 1.0 | Sequential disk page fetch |
| random_page_cost | 4.0 | Random disk page fetch |
| cpu_tuple_cost | 0.01 | Processing each row |
| cpu_index_tuple_cost | 0.005 | Processing each index entry |
| cpu_operator_cost | 0.0025 | Evaluating each operator |
| parallel_setup_cost | 1000 | Launching parallel worker |
| parallel_tuple_cost | 0.1 | Tuple transfer between processes |
| effective_cache_size | 4GB | Assumed disk cache available |

## Statistics Types

### Single-Column Statistics
1. **Histograms**: equi-depth (equal frequency) buckets
   - PostgreSQL: default 100 buckets
   - Configurable per column via ALTER TABLE SET STATISTICS
2. **Most Common Values (MCV)**: top-K frequent values + frequencies
   - Critical for skewed distributions
   - PostgreSQL: stored in pg_stats
3. **N-distinct**: estimated number of distinct values
   - Negative values indicate fraction of rows (e.g., -1.0 = unique)
4. **Null fraction**: fraction of NULL values
5. **Correlation**: physical vs logical ordering correlation

### Multi-Column Statistics (Extended)
1. **Functional dependencies**: column A determines column B
   - Fixes independence assumption for correlated columns
2. **N-distinct combinations**: distinct count for column groups
   - Improves GROUP BY cardinality estimation
3. **Multivariate MCV**: common value combinations
   - Addresses correlated predicates (city, state)

### Sketches and Sampling
1. **Count-Min Sketch**: approximate frequency estimation
2. **HyperLogLog**: approximate distinct count
3. **Reservoir sampling**: random sample for on-the-fly estimation

## Calibration Techniques

### Static Calibration
- Run micro-benchmarks to measure actual hardware costs
- Set cost parameters to match observed I/O and CPU ratios
- Re-calibrate when hardware changes

### Dynamic Calibration
- Compare estimated vs actual costs after execution
- Adjust cost parameters over time
- Feedback loop: EXPLAIN ANALYZE captures actuals

### Auto-Tuning
- PostgreSQL: manual (DBA sets parameters)
- CockroachDB: some automatic cost model adjustment
- ML-based: learned cost models (recent research)

## Common Pitfalls

1. **Cardinality estimation errors compound**: 2x error per join -> 2^N error for N joins
2. **Independence assumption**: P(A AND B) != P(A) * P(B) for correlated columns
3. **Uniform distribution assumption**: real data is skewed
4. **Stale statistics**: ANALYZE not run after major data changes
5. **String statistics**: histograms on strings are less accurate
6. **Function selectivity**: optimizer often uses magic constants (0.33 for LIKE)
7. **Correlated subqueries**: cardinality estimation nearly impossible
8. **Join selectivity with complex predicates**: only equi-joins well estimated

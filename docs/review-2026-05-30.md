# Ra Review — 2026-05-30 (planner_hook fixes + PG19 bring-up)

Reviewer: gregburd (with AI assistance). Same process and threat
model as `docs/security-review-2026-05-29.md`: Ra runs **in-process
inside a PostgreSQL backend** via `planner_hook`, so any Ra defect is
at minimum a backend-crash DoS, and — as this review found — at worst
a source of **silent wrong query results**.

Scope: the changes made while fixing the two pre-existing
plan_builder bugs reported on 2026-05-29, plus the PostgreSQL
19-devel bring-up (forked pgrx on `feature/pg19-devel`, Ra extension
on `feature/pg19`). Verified end-to-end against a local assert-enabled
PostgreSQL 19devel cluster.

## What changed

Fixes (on `main`, version-independent; mirrored on `feature/pg19`):

1. **planner_hook crash-safety (critical).** The `catch_unwind` `Err`
   arm raised `pgrx::error!`. `ra_planner_hook` has no `#[pg_guard]`
   boundary, so that error raises a `panic_any` that cannot unwind
   across PostgreSQL's C `planner()` frame and **aborts the backend**
   (`fatal runtime error: failed to initiate panic`). Now the arm
   degrades to the native planner (`call_prev_planner`) — the core
   invariant from the prior review: *never crash a query PG can
   plan*. The fallback message is decoded from pgrx
   `CaughtError`/`ErrorReportWithLevel` payloads and gated behind
   `ra_planner.log_decisions`.
2. **`PlannedStmt.permInfos` (pg16+).** Ra copied the parse Query's
   `rtable` (with `perminfoindex` values intact) but never set
   `permInfos`, so the executor rejected plans with *"invalid
   perminfoindex N"*. Now propagated. **Fixes reported bug #1.**
3. **`PlannedStmt.unprunableRelids` (pg18+).** Ra emits no run-time
   pruning, so all range-table entries are unprunable; set them, or
   pg18+ raises *"trying to open a pruned relation."*
4. **`stats_bridge` OID-list accessor.** `RelationGetIndexList`
   returns an OID list; the code used `list_nth` (asserts
   `IsPointerList`) which aborts on an assert build during planning.
   Switched to `list_nth_oid` + a NIL guard. **This — not the
   feedback path — was the true cause of reported bug #2 ("EXPLAIN
   ANALYZE aborts"); EXPLAIN ANALYZE now runs.**
5. **`feedback_hook` panic guard.** `capture_feedback` wrapped in
   `catch_unwind` so executor telemetry can never abort the backend.

Also: the messy commit `90018349` (which bundled the standards-file
deletions under a "satisfy clippy" subject) was split via rebase into
an accurate clippy-fix commit and a `refactor: remove dead
grammar/standards scaffolding` commit, and force-pushed (authorized).
Verified the rewritten tree is byte-identical to the pre-rewrite tree
(`git diff backup-main-pre-rewrite HEAD` empty).

## Verified

- The two reported bugs are fixed: zero *"invalid perminfoindex"*
  errors; EXPLAIN ANALYZE produces output without aborting.
- No configuration of any tested query produces a **hard backend
  crash** anymore — the planner-hook and feedback-hook guards contain
  every panic and fall back.
- The full pgrx pg19 fork compiles, installs, and loads; all
  `ra_planner.*` GUCs register; `noidx`/scan/filter/join shapes have
  each returned correct results in at least some runs.

## Findings — CRITICAL, not fixed

Re-testing past the crashes exposed two deeper, pre-existing defects.
These are **more serious than the two reported bugs** and mean
**Ra in-process planning is not safe or correct on PG19 today.**

### A. Heap double-free in the index-stats path (critical)

Planning a query over a table **with an index** intermittently hits
`pfree called with invalid pointer ... (header 0x7f7f7f7f7f7f7f7f)` —
the `0x7f` fill is PostgreSQL's freed-chunk clobber pattern, i.e. a
**double-free / heap corruption** originating in
`stats_bridge`'s index-stats gathering (a `Drop` after planning; the
offending chunk is deterministic but the trigger is not). PG's
`pfree` guard *sometimes* catches it (→ clean fallback, correct
result) and *sometimes* does not (→ silent corruption). It was
localized to the index path (the obvious sites — `resolve_am_type`'s
`pfree`, `list_free`, `register_feedback` — were each ruled out) but
**not root-caused**. The crash-safety fix prevents the hard abort but
**cannot prevent the silent corruption.**

### B. Non-deterministic / wrong results when Ra's plan executes (critical)

When Ra's own plan runs (no fallback), results are
**non-deterministic and frequently wrong** — e.g.
`SELECT id, v FROM t WHERE id = 42` returned all 1000 rows (WHERE
filter effectively dropped) on one run and 1 row on another; the
outcome flips with unrelated changes (e.g. toggling
`log_decisions`, which perturbs heap allocation). This is consistent
with finding A corrupting plan structures, but a genuine
plan-translation defect (quals not always carried into the emitted
`Plan`) cannot be ruled out. Either way, **Ra can silently return
incorrect results.**

## Recommendation

- **Do not enable `ra_planner.enabled` for real workloads** until A
  and B are resolved. Given the silent-wrong-results risk, consider
  defaulting the GUC to `off`.
- Root-cause A with a memory debugger (the masked Rust panic +
  PG's repeated-crash report throttling defeated `eprintln`/crash-log
  inspection here; ASan/valgrind or a `pfree`-instrumented PG build is
  the right next tool).
- Add a differential-correctness gate (the `ra-difftest` harness)
  comparing Ra-on vs Ra-off result sets across query shapes before
  re-enabling — it would have caught B immediately.

## Honest residuals

- `feature/pg19` depends on the unpublished pgrx fork via a path
  dependency; it is a bring-up branch, not mergeable to `main` as-is.
- The pg19 ABI is assumed equal to pg18 wherever pgrx gates on
  version; validated against current PG19-devel HEAD only — it may
  drift before release.
- `ra-pg-extension` carries pre-existing clippy lint debt (it is
  out-of-workspace with relaxed lints); untouched here.

# Re-review — 2026-05-30 (resolution of defects A & B)

Reviewer: gregburd (with AI assistance). Follow-up after fixing the
two critical defects above. Verified on PostgreSQL 19-devel
(`feature/pg19`); fixes cherry-picked to `main` and compiled against
pg18.

## Root causes and fixes

**Defect B (wrong results) — root-caused and fixed.** Not
non-determinism in the optimizer: Ra's Lime parser emits *unqualified*
column references (`col.table = None`) for `SELECT col FROM t`, and
`expr_translator::column_to_var` keyed its lookup on table name, so an
unqualified column produced a **NULL `Var`**. That silently dropped
targetlist entries and filter quals — `SELECT a,b FROM t WHERE a=42`
returned every row with a blank column. Fix: resolve unqualified
columns against the range-table relation that owns the column (PG's
own name-resolution behavior). The apparent "non-determinism" was the
heap glitch (A) sometimes triggering a fallback that masked the wrong
plan.

**Defect A (heap double-free) — contained at source.** An intermittent
`pfree`-of-already-freed-pointer in index-stats gathering. PG's
`pfree` guard detects the bogus header and raises before any actual
corruption, so it never silently corrupted memory; the prior crash was
the recovery path, already fixed. It is now wrapped in `catch_unwind`
in `gather_index_stats`, so a glitch degrades to "no index stats for
this relation" instead of failing the plan. Not fully root-caused (it
became too rare to capture a backtrace after B was fixed — ~1 in
hundreds of plans), but it is no longer reachable as a crash or a
wrong result.

## Additional safety work

- **Correctness gate** (`PlanBuilder::is_translatable`): Ra emits a
  plan only for verified shapes — single-relation `Scan` / `Filter` /
  `Project`. Aggregate, Join, Sort, Limit, Distinct, set-ops, DML
  defer to the native planner. Found while re-testing that those
  builders variously drop rows, leave `aggregate_dummy` unresolved,
  corrupt executor memory (ORDER BY/LIMIT), or crash the backend
  (DISTINCT). Gating them off makes `ra_planner.enabled = on` correct
  by construction.
- **Scan-strategy advice no longer physically honored.**
  `BITMAP_HEAP_SCAN` advice crashed the backend; IndexScan/Tid
  builders are unverified. Advice is still parsed/validated but every
  `Scan` is emitted as a `SeqScan`. The crashing Filter peephole was
  removed.

## Verification — the property test the brief asked for

`scripts/replan-equivalence-test.sh` replans one fixed query under the
cross product of `ra_planner.plan_advice` ∈ {none, SEQ_SCAN,
BITMAP_HEAP_SCAN, TID_SCAN, INDEX_SCAN} and three statistics states,
asserting the result multiset is identical to the native planner for
every variation (execution time is not checked). It caught the
`BITMAP_HEAP_SCAN` crash. After the fixes: **15/15 replans
row-identical**, and a 60-query mixed stress run showed **0 crashes, 0
memory-safety events, 0 panics**. Ra demonstrably optimizes the
scan/filter/projection shapes (`ra_planner: OK` in the decision log),
and every other shape defers to PG.

## 1.0 posture — honest

- **Reliable: yes, by construction.** Every plan Ra emits is verified
  row-equivalent to PG; everything else falls back. `ra_planner.enabled
  = on` will not return wrong results or crash the backend.
- **Performant: partial.** Ra currently optimizes only single-relation
  scan/filter/projection queries; real workloads dominated by joins
  and aggregates fall back to PG and see no Ra speedup. Reaching
  "performant on real production workloads" requires implementing and
  verifying the Join / Aggregate / Sort / Limit plan builders, each
  admitted to `is_translatable` only after it passes the
  replan-equivalence test. That is the path from this reliable
  baseline to a full 1.0; it is scoped future work, not a regression.
